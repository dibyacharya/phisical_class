4. Lumens VC-TR1 Camera — Capability Comparison

Camera Deep-Dive

The Lumens VC-TR1 is an AI-powered PTZ (Pan-Tilt-Zoom) camera with auto-tracking, presets, RTSP streaming, NDI broadcast, and full network management. It is a flagship-class camera designed for professional production environments — broadcast studios, telemedicine, courtrooms, classrooms.

This is a substantial investment. The question is: how much of its capability can each platform actually use?

4.1 Lumens VC-TR1 — Full Feature Set (As Designed)

  • AI Auto-Tracking: Camera follows the active speaker automatically using face and voice detection.
  • PTZ Mechanism: 12× optical zoom, 340° pan, 120° tilt. Smooth motorised movement.
  • Preset Positions: Up to 256 saved positions, instantly recallable (e.g., "wide shot", "teacher close-up", "blackboard").
  • Dual Output: Simultaneous USB + Ethernet RTSP — feed two systems at once.
  • Network Streaming: RTSP, NDI 5, RTMP, SRT — all major broadcast protocols.
  • Web Management: Browser-based admin from any computer on the network.
  • Multi-Bitrate Output: Camera can output multiple resolution/quality streams concurrently.
  • API Control: Full programmatic control over PTZ, presets, recording, and tracking modes.
On Windows: 100% of features unlocked
  • ✓ AI auto-tracking — works directly
  • ✓ PTZ control — programmable presets
  • ✓ RTSP Ethernet streaming — native input to our Windows recording application
  • ✓ NDI streaming — plugin available
  • ✓ Web admin GUI — Chrome/Edge access
  • ✓ Multi-stream output — both USB + IP
  • ✓ Python/REST API control — automate everything
  • ✓ Firmware updates — Lumens utility
  • ✓ Centralized fleet management

Result: Camera operates as the manufacturer designed it. Investment fully realized.

On Android: ~30% of features available
  • ✗ AI auto-tracking — no API path
  • ✗ PTZ control — Android USB API has no PTZ
  • ✗ RTSP streaming — no built-in client
  • ✗ NDI streaming — Windows-only ecosystem
  • ✗ Web admin GUI — no usable browser on TV
  • ✗ Multi-stream — Android sees one USB stream
  • ✗ API control — no path to camera's API
  • ✗ Firmware updates — must remove + connect to PC
  • ✓ Basic USB video capture (single stream)

Result: Camera operates as a basic USB webcam. ~70% of paid-for capability dormant.

Investment Realization
The Lumens VC-TR1 has a market price reflecting its AI, PTZ, and IP capabilities. On Android, the system uses it as a plain webcam — equivalent to a ₹5,000 USB camera. On Windows, the full investment value is realized.

4.2 What "Full Automation" Actually Means

"Full automation" with the Lumens camera on Windows means scenarios like:

  • Class auto-start: Schedule arrives, system commands camera to "Lecture Mode" preset (wide shot of room), starts recording. Zero teacher action.
  • Speaker tracking: AI detects who's speaking, camera follows automatically. No camera operator needed.
  • Q&A handover: When student asks question from back of room, camera auto-pans to focus on them.
  • Blackboard mode: Teacher walks to blackboard, camera switches to "Whiteboard" preset showing the board area.
  • Multi-angle recording: Camera outputs two streams — one for live cloud view, another for high-quality recording.

None of the above is possible on Android, because Android's USB Camera API exposes only basic video frames — no commands, no presets, no AI control surface.

DHUUN Team Validation
The DHUUN team — the AV systems integration partner who supplied and configured the Lumens VC-TR1 — has confirmed that the camera's full feature set (AI auto-tracking, PTZ control, RTSP streaming, NDI output, multi-bitrate output, and centralized management) is operationally accessible on Windows-based platforms. On Android-based recording, the camera operates only in basic USB video mode; its advanced features cannot be invoked due to operating-system-level API limitations on Android.

How It Works Today

6.2 The Core Engineering Challenge

The signage TV's processor is designed for displaying digital signage content — relatively low-load tasks. Continuous recording with screen capture, camera capture, audio mixing, software encoding, and live network upload pushes the SoC well beyond its design envelope.

Within minutes of recording, the CPU heats up. Without intervention, this leads to two outcomes: either the system crashes, or the SoC silently downclocks and starts dropping frames. Neither is acceptable for classroom recording.

To keep the system stable, we engineered a custom Thermal Governor — software that monitors CPU usage and chip temperature in real-time, and takes protective action before the system fails.

6.3 The Thermal Governor — A 5-Tier State Machine

┌─────────┐ stress ┌────────┐ stress ┌──────────┐ stress ┌──────────┐ │ NORMAL ├────────────►│ WATCH ├────────────►│ THROTTLE ├─────────────►│ COOL │ └─────────┘ └────────┘ └─────┬────┘ └──────────┘ ▲ ▲ │ │ │ │ ▼ │ │ │ ┌──────────────────┐ │ │ │ │ Pause camera 45s │ │ │ │ └────────┬─────────┘ │ │ │ │ │ │ └───────────────────────┘ │ │ cooling │ │ │ └───── recovery (CPU<80%, 90s, temp<67°C) ──────────────────────────────┘

The system continuously monitors CPU and temperature, and transitions through five tiers based on observed stress:

Tier Trigger What Happens
NORMAL CPU < 90% Full recording — camera, screen, audio, all active.
WATCH CPU ≥ 90% sustained for 60 seconds Increase logging verbosity. No user-visible change. System is alerted.
THROTTLE CPU ≥ 95% sustained for 5 minutes OR chip temp ≥ 70°C Camera is paused for 45 seconds to let the SoC cool. Recording continues with screen + audio only. Camera then auto-resumes.
COOL CPU ≥ 98% sustained for 2 minutes OR chip temp ≥ 78°C (emergency) Extended camera pause (~90 seconds) to give SoC more cooling time. Recording continues without camera.
ABORT Critical hardware fault Recording stops gracefully, MP4 finalized, alert sent to admin.

6.4 The Cycling Camera Behavior (v3.7.3 — Current)

In real classroom use, the system frequently enters THROTTLE state during 1-hour classes because of sustained CPU pressure. Earlier versions of our software would drop the camera permanently in this case (resulting in an audio-only recording for the rest of class).

In the current version (v3.7.3), we engineered a cycling camera behavior that maximizes camera-on time:

Class starts Class ends │ │ ▼ ▼ NORMAL ──► THROTTLE ──► (45s pause) ──► NORMAL ──► THROTTLE ──► (45s pause) ──► NORMAL ──► ... ▲ │ ▲ │ │ ▼ │ ▼ │ Camera OFF │ Camera OFF │ Audio + Screen │ Audio + Screen │ continue recording │ continue recording │ │ └─── Camera back ON when CPU < 80% ──────┘ for 90 seconds and temp < 67°C (whichever condition met first)

This cycling logic means: instead of losing camera permanently after the first overheat event, the camera comes back as soon as the SoC cools enough. In practice, this gives the client substantially more camera footage during long classes.

However — A Fundamental Limit Remains
Even with the cycling camera behavior, there are still 45-second to 90-second windows during a 1-hour class where the camera feed is dropped. Audio and screen capture continue, but the visual record has gaps. This is a hardware limitation that cannot be fully eliminated by software — the SoC is simply not powerful enough to sustain full recording load for 60 minutes without thermal management.

6.5 What the Thermal Governor Does NOT Solve

  • Hardware encoder limitations: The TV's video encoder is shared with display rendering. Even at idle, encoding 1080p continuously consumes most of its capacity.
  • USB bandwidth bottleneck: USB 2.0 ports limit camera+mic bandwidth; no way to expand.
  • Inability to use camera/mic IP features: Android USB stack does not expose RTSP, Dante, PTZ, or DSP features.
  • Storage I/O slowness: eMMC storage is ~5× slower than NVMe SSD; impacts buffer flushes during peak load.
  • Memory pressure: 2GB RAM is tight when running encoder + screen capture + live streaming publisher + UI.
  • Update mechanism complexity: Required us to build a custom OTA system, which adds engineering maintenance overhead.

7. Architecture Transition — Current to Proposed

System Architecture

This section illustrates how the system structure transitions from the current Android-based architecture to the proposed Windows-based architecture, and how the Windows architecture enables reliable concurrent multi-room recording at scale.

7.1 Current Architecture (Android-Based)

┌──────────────────────────────────────────────────────────────────┐ │ LG 55TR3DK Smart Signage TV │ │ │ │ USB ◄── Camera (Lumens VC-TR1) — basic USB mode only │ │ USB ◄── Mic (Sennheiser TCC2) — basic USB audio mode only │ │ │ │ ▼ │ │ Built-in Android SoC │ │ • Software encoder (CPU-based) │ │ • Custom thermal management │ │ • Live stream publisher │ │ ▼ │ │ [Ethernet] ──► Our cloud streaming + recording infrastructure │ └──────────────────────────────────────────────────────────────────┘ Recording path: Camera → TV SoC → Encoder → Live Stream → Cloud Recording Display path: TV's Android UI on the same TV screen.

Bottlenecks observed in current architecture

  • The TV SoC handles capture, encode, and upload simultaneously — the device overheats during long recordings
  • Camera and microphone are limited to USB basic modes; their advanced features (RTSP, Dante, PTZ, beamforming) are not reachable from Android
  • The cloud recording component is shared across all rooms — concurrent recording capacity must be scaled centrally
  • If the TV reboots or the Android app crashes, the recording pipeline must restart entirely
  • Recording quality is directly tied to network reliability at the moment of class

7.2 Proposed Architecture (Windows-Based)

┌──────────────────────────────────────────────────────────────────┐ │ Windows Mini PC │ │ │ │ Ethernet ◄── Camera (Lumens VC-TR1) — RTSP, full features │ │ Ethernet ◄── Mic (Sennheiser TCC2) — Dante, full features │ │ │ │ ▼ │ │ Our Windows Recording Application (D&R AI Solutions) │ │ • Hardware video encoder (Intel Quick Sync) │ │ • Multi-source compositing (camera + screen + audio) │ │ • Local recording to SSD — always-on, network-independent │ │ • Live streaming via our video conferencing platform │ │ • Background cloud upload — queued, resumable │ │ ▼ │ │ ┌──────────────────────┐ ┌──────────────────────────────┐ │ │ │ Local SSD recording │ │ Live stream → our cloud │ │ │ │ (primary, immediate) │ │ (only when remote view used) │ │ │ └──────────┬───────────┘ └──────────────────────────────┘ │ │ │ │ │ ▼ background upload │ │ [Ethernet] ──► Our cloud archive │ │ │ │ Display ──HDMI──► LG TV (used purely as a monitor; │ │ recording does not depend on it) │ └──────────────────────────────────────────────────────────────────┘ Recording path: Camera → Mini PC → Hardware encoder → Local SSD ─┐ │ Background upload ─▼ Cloud Archive

Why this architecture works better

  • Recording happens on dedicated computing hardware, not on display hardware
  • Camera and mic use their full-featured Ethernet/Dante paths instead of basic USB
  • Local SSD recording is the source of truth — network interruption does not affect the recording itself
  • Each Mini PC operates independently — no shared real-time bottleneck across rooms
  • Live streaming becomes optional — used only when a remote viewer needs to watch the class live

Multi-Room Capacity
With the Windows architecture, recording capacity scales linearly with the number of rooms — each Mini PC contributes its own complete recording capability. There is no shared real-time recording bottleneck. The cloud only handles background upload of completed recordings, which is an asynchronous batch operation that scales straightforwardly.

8. Smart Notebook & Teaching Tools

Teacher Continuity

A reasonable concern when changing the platform is whether teachers will face disruption to their existing workflow. Smart notebook applications, annotation tools, slide presentation, and interactive whiteboard features must continue to be available — and ideally, become richer.

Both Android and Windows platforms support smart notebook applications. Windows, however, has a substantially larger and more mature ecosystem in this category, particularly in the education segment.

8.1 Smart Notebook Application Availability

Android Smart Notebook Apps
  • Apps available through the Android app ecosystem, subject to the signage TV's app installation policy
  • Touch interaction depends on the TV's touch overlay (where present)
  • Annotation, drawing, and basic save/share are supported by mainstream apps
  • Cloud sync via Google or app-specific backends
  • Stylus pressure support is limited — most signage TVs are not pen-enabled by design

Functional, but constrained by signage TV's app ecosystem and input hardware.

Windows Smart Notebook Apps
  • Microsoft Whiteboard — free, cloud-synced, multi-user collaboration
  • Microsoft OneNote — full stylus, ink-to-math, OCR, Office integration
  • SMART Notebook — long-established standard in education classrooms
  • Promethean ActivInspire — established interactive whiteboard software
  • Open-source options — OpenBoard, Open-Sankoré (free, classroom-focused)
  • Web-based options — Jamboard, Miro, Mural, FigJam (browser, no install)
  • Native Windows Ink API — hardware-accelerated drawing, palm rejection
  • Stylus support across compatible touch displays with pressure sensitivity

Mature, broad ecosystem with category-defining tools used across global education.

8.3 Teacher Onboarding

For teachers transitioning from Android-based notebook apps to Windows-based notebook apps, the user experience is straightforward:

  • Drawing and annotation interactions are visually consistent across both platforms — pinch to zoom, drag to pan, tap to write.
  • Windows notebook apps are widely used in education globally — teachers may already be familiar with these from previous institutions, training programs, or personal use.
  • Stylus and touch input on Windows are mature — Windows Ink has been a core operating-system feature since 2016 and is the standard input model for educational tablets and pen displays.
  • Notebook content integrates naturally with the lecture recording — anything drawn on the notebook becomes part of the recorded lecture without separate handling.

The optimizations above can extend the current system's useful life and improve reliability marginally. None of them, however, can unlock the Lumens or TCC2 hardware features that require RTSP, Dante, or PTZ control — those capabilities are blocked at the Android operating system level, not by our software. Squeezing more performance from Android improves the same plateau; it doesn't move us to a new one.

9.3 Could the Current System Run More Smoothly at 720p?

A practical question is whether reducing the recording quality from 1080p to 720p would allow the existing Android-based system to operate more reliably for 1-hour classes. This is a meaningful question because lower resolution means lower encoding workload, and lower encoding workload means less thermal stress on the TV's SoC.

What 720p Would Improve

  • 1-hour classes would likely complete with significantly fewer or zero camera-drop cycles
  • Lower SoC heat — thermal throttle state reached less often, if at all
  • Faster cloud upload, less network bandwidth consumed
  • Less storage pressure on the TV's eMMC (which is slow)
  • Reduced risk of memory pressure during long classes

What 720p Does NOT Solve

  • Lumens VC-TR1 advanced features remain unreachable — AI auto-tracking, PTZ control, RTSP, NDI, multi-bitrate streaming. These are blocked at the Android operating system level, not by quality settings.
  • Sennheiser TCC2 advanced features remain unreachable — beamforming, Dante, multi-channel routing, DSP control. Same reason.
  • Multi-room concurrent capacity still constrained by shared cloud recording.
  • 2-hour+ continuous classes still thermally challenging — the SoC is fundamentally not designed for sustained recording loads of this length.
  • Custom OTA system, custom thermal management, custom recovery logic all still required.
Engineering Verdict on 720p
Reducing recording quality to 720p would meaningfully improve thermal stability for 1-hour classes, with fewer or zero camera-drop cycles per class. However, this comes at the cost of recording detail — particularly for board-heavy and slide-heavy lectures where small text becomes harder to read on playback. Crucially, 720p does not unlock any of the Lumens or Sennheiser advanced features, and does not address platform-level limitations such as multi-room concurrent capacity or 2-hour class viability.

10. What Is Uniquely Possible on Windows

Windows Capabilities

Windows is not just "a different platform" — it is the platform that the Lumens and TCC2 hardware was designed for. Here are capabilities that become available on Day 1 of a Windows deployment.

10.1 Day-1 Out-of-the-Box Capabilities

★ Headlining Feature: Full Camera/Mic Automation
Windows can speak directly to the Lumens camera (RTSP, API) and TCC2 mic (Dante, API). The system can automatically position the camera based on class type, follow the speaker via AI, switch to blackboard view, and steer microphone beams — without any manual intervention. None of this requires custom development; it uses standard manufacturer integration.
★ Recording Independence (Decoupled Architecture)
On Windows, recording happens on the Mini PC. The display is a separate concern. Even if the display fails, the user logs out, or the screen is unplugged, recording continues. Power outage with UPS — recording resumes without data loss. This is the architectural pattern used by professional broadcast studios.
★ Multi-Camera Support
Add a second camera (front of room + back of room, or main + document camera) at any time. Our Windows recording application supports multi-camera scene switching natively. Currently impossible on Android.
★ Concurrent Stream Output
Same recording can simultaneously: (a) save to local disk, (b) upload to cloud, (c) live-stream to Zoom/Meet/YouTube. All from the same Mini PC, in parallel. No additional infrastructure.
★ Standard IT Operations
No more custom OTA system to maintain. Windows Update, Group Policy, Active Directory, BitLocker, antivirus, RDP — all the tools your IT team already knows. Onboarding new IT staff is trivial; signage TV management is a niche skill.
